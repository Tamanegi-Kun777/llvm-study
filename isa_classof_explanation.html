isa と classof の仕組み

9ccの kind(enumの値)による判定と比較しながら、DummyC(LLVM)の isa / classof を整理する。

前提のおさらい:なぜこの仕組みが必要なのか

9ccでは、すべてのノードが同じ Node という1つの構造体だった。 

struct Node {
  NodeKind kind;  // ND_ADD, ND_IF, ND_NUM ...
  Node *lhs;
  Node *rhs;
  ...
};

「今扱っているノードが何の種類か」を確認するには、node->kind というフィールドを直接見るだけでよかった。 

if (node->kind == ND_ADD) { ... }

しかしDummyCでは、BinaryExprAST, NumberAST, VariableAST のように種類ごとに別々のクラスを作り、 それぞれが BaseAST という共通の親クラスから継承する、という設計を取っている。 

BaseAST *node = ...;   // これは、本当は BinaryExprAST かもしれないし、NumberAST かもしれない

node という変数は BaseAST* という型なので、見た目だけではその先が実際には何のクラスなのか分からない。 ここで「今この node は実際には BinaryExprAST ですか」という確認をするための仕組みが isa と classof。

isa とは何か

llvm::isa<T>(ポインタ) は、LLVMが用意している「このポインタは本当に T という型か」を確認する便利な関数。 

if(llvm::isa<BinaryExprAST>(stmt)){
  // stmt は、本当は BinaryExprAST だった
}

これは9ccで言えば 

if (node->kind == ND_ADD || node->kind == ND_SUB || ...) {
  // これは、二項演算ノードだった
}

に相当する。「stmt という BaseAST* のポインタが、実は BinaryExprAST というより具体的な種類だったか」を true/false で教えてくれる。

isa は内部で何をしているのか(ここが本題)

isa<T>(ポインタ) は、実はプログラマが書いた classof という関数を代わりに呼び出しているだけ。以下の2つは実質的に同じ意味になる。 

llvm::isa<BinaryExprAST>(stmt)
BinaryExprAST::classof(stmt)

つまり isa 自体は「どうやって判定するか」を知らない。判定の具体的な中身は、各クラスが自分で classof という関数として用意しておく必要がある。 isa は、その classof を呼び出す共通の窓口にすぎない。

classof をもう一度、丁寧に

class BinaryExprAST : public BaseAST
{
  ...
public:
  // BinaryExprASTなのでtrueを返す
  static inline bool classof(BinaryExprAST const*){return true;};

  // 渡されたBaseASTがBinaryExprASTか判定する
  static inline bool classof(BaseAST const* base){
    return base->getValueID() == BinaryExprID;
  };
};

classof は2つ用意されていることに注目。それぞれ役割が違う。

1つ目: classof(BinaryExprAST const*)

static inline bool classof(BinaryExprAST const*){return true;}

「もし渡されたポインタがすでに BinaryExprAST* 型として宣言されているなら、それは当然 BinaryExprAST です。無条件に true を返します」という当たり前の判定。 実際にはあまり使われないが、isa の仕組みが正しく動くために形式的に必要とされている。

2つ目: classof(BaseAST const* base)

static inline bool classof(BaseAST const* base){
  return base->getValueID() == BinaryExprID;
}

こちらが実際に使われる本命の判定。「渡された BaseAST*(親クラスのポインタとして扱われているもの)の getValueID()(IDというメンバー変数を返す関数)が、 BinaryExprID(enum AstID の値)と一致するか」を確認している。

BaseAST の ID というメンバー変数を、もう一度確認

class BaseAST
{
  AstID ID;
public:
  BaseAST(AstID id) : ID(id){};
  AstID getValueID() const {return ID;};
};

すべての BaseAST から継承したクラスは、コンストラクタで自分の種類を ID という変数に記録しておく。 

BinaryExprAST(...) : BaseAST(BinaryExprID), ...{};

BinaryExprAST が作られるその瞬間に、「私は BinaryExprID という種類です」と親クラス(BaseAST)の ID に記録しておく、というわけ。 これは9ccの 

node->kind = ND_ADD;

と全く同じ役割。classof は、この記録された ID を確認するだけ。

流れの図

1. BinaryExprAST が、作られるとき → BaseAST(BinaryExprID) が呼ばれる → ID という変数に、BinaryExprID が記録される (9ccの node->kind = ND_ADD; と同じ) 2. isa<BinaryExprAST>(stmt) が、呼ばれる → 内部で、BinaryExprAST::classof(stmt) が、呼ばれる → stmt->getValueID() が、ID を返す → その ID が、BinaryExprID と、一致するか、確認する (9ccの node->kind == ND_ADD と同じ) 3. 一致すれば true、しなければ false が、isa の戻り値になる

dyn_cast との関係

isa とセットでよく使われる dyn_cast にも触れておく。

BinaryExprAST *bin = llvm::dyn_cast<BinaryExprAST>(stmt);

dyn_cast<T>(ポインタ) は、「もし isa<T>(ポインタ) が true なら、そのポインタを実際に T* 型に変換して返してください」という意味。 内部で isa(つまり classof)をまず確認してから変換を行っている。もし種類が違えば NULL が返ってくる。

9ccでは、こういった「安全な型変換」という考え方自体が無かった。9ccでは Node* という1種類のポインタしか無かったので、変換の必要がそもそも無かった。

まとめ

用語役割9ccとの対応
BaseAST::ID 「今このノードが何の種類か」を記録するメンバー変数 node->kind
classof 「渡されたポインタがこのクラスかどうか」を ID を見て判定する、各クラス自身が用意する専用の関数 node->kind == ND_ADD という比較そのもの
isa<T>(ポインタ) T::classof(ポインタ) を代わりに呼び出してくれる共通の窓口関数 if (node->kind == ND_ADD) の書き方を統一したもの
dyn_cast<T>(ポインタ) isa で確認した上で、実際に型を変換して返してくれる便利関数 (9ccには対応する概念が無い。型が1種類だったため)

isa と classof は、「9ccが kind という1つの変数の比較で済ませていたことを、C++のオブジェクト指向の流儀に沿って、 各クラスが自分自身で判定の仕方を提供する」という形に置き換えた仕組みである。LLVM自身がこのパターンを標準的な作法として広く使っているため、DummyCもそれに倣っている。
